# Job: Ansible Lint Validation

## Overview

This job demonstrates how to use **ansible-lint**, a deterministic rule-based tool, to validate Ansible playbooks for best practices, security issues, performance problems, and style violations.

Unlike AI-based analysis (like `job1-code-scan`), ansible-lint provides:
- ✅ **Deterministic Results** - Same playbook always produces same findings
- ✅ **100+ Built-in Rules** - Comprehensive coverage of best practices
- ✅ **Structured Output** - JSON format for easy parsing
- ✅ **Fast Execution** - No LLM processing overhead
- ✅ **Industry Standard** - Official Ansible recommendations

## Job Files

### `agent/`
- **`instruction.md`** - Task definition for the agent (what to do with the findings)
- **`SKILL-ansible-lint.md`** - Documentation on the ansible-lint tool and how to use it

### `inbox/` (Sample Playbooks)
- **`playbook-clean.yaml`** - Well-structured, passes all checks
- **`playbook-style-issues.yaml`** - Formatting and style violations
- **`playbook-violations.yaml`** - Security and best practice violations

### `outbox/`
Generated by the agent:
- `lint_summary.md` - Executive summary
- `findings_by_severity.json` - Categorized violations
- `remediation_guide.md` - How to fix each issue
- `playbook_reports/` - Individual lint reports per file
- `improvement_roadmap.md` - Priority order for fixes
- `compliance_score.txt` - Current compliance percentage

## How It Works

1. **Agent receives playbooks** from the inbox
2. **Agent runs ansible-lint** on each file using the SKILL-ansible-lint guidance
3. **Agent parses results** and categorizes by severity and rule
4. **Agent generates reports** with actionable remediation steps
5. **Results written to outbox** for review

## Understanding ansible-lint Rules

### Rule ID Format
- **E### (Errors)** - Critical issues that must be fixed
- **W### (Warnings)** - Best practice violations, should be fixed
- **I### (Info)** - Informational, nice-to-fix

### Common Rules in Examples

| Rule | Severity | Issue | Example |
|------|----------|-------|---------|
| E602 | LOW | No tabs (use spaces) | Tab character in YAML |
| E704 | HIGH | Use shell only when necessary | `shell: mkdir -p` instead of `file:` |
| W502 | HIGH | Deprecated module | Using `docker_service` instead of containers module |
| E701 | MEDIUM | run-once with loop | Conflicting directives |
| E401 | HIGH | Unexpected jinja2 | Jinja2 syntax in wrong context |

## Using ansible-lint Locally

If you have ansible-lint installed:

```bash
# Scan a playbook
ansible-lint playbook.yaml

# Scan with JSON output
ansible-lint playbook.yaml --format json

# Scan entire directory
ansible-lint roles/ --format json

# Show all available rules
ansible-lint --help-list

# Fix auto-fixable issues
ansible-lint --fix playbook.yaml
```

## Integration with CI/CD

Use ansible-lint in your pipeline:

```yaml
# GitHub Actions example
- name: Lint Ansible playbooks
  run: |
    pip install ansible-lint
    ansible-lint playbooks/ --strict
```
